# KBA-502 — Summary of Risk Assessments (the risk INPUT form — the IR/CR/RMM write target)

**Form:** KBA-502 (`dataBindingKey` KBA502). Uniform across the five non-SA titles.
**Role in the cascade:** the risk **input**. For each relevant assertion selected on KBA-400, the firm's
**Inherent Risk (IR)**, **Control Risk (CR)**, derived **RMM**, a **link to controls testing** when CR < MAX,
and the **planned audit approach** (Combined / Substantive-Analytical / Substantive-In-depth) are assessed
and **written HERE, on KBA-502**. The AUD-8xx program grids read these values through.

## Data model

**KBA-502 OWNS the per-assertion IR/CR/RMM/approach grid and is the WRITE target.** The AUD-8xx program's
`.{AREA}.RelevantAssertion` grid is the **DERIVED / read-through** view. Writing the grid against a
program's workpaperId lands in the program's working copy, which the
KBA-502-owned recompute **discards on refresh**.

- KBA-502's `elements` define the grid columns `OverallAuditAreas[current].RelevantAssertion` →
  C03 SELECTED · C04 IR · C05 CR · C06 RMM · C07 WPREFERENCE · C08 PLANNEDAUDITAPPROACH · C09
  PROGRAMSTEPS · C10 USERCOMMENTS — structurally identical to the program grid.
- The UI, setting an IR radio on KBA-502, fires `UpdateProperty` against **KBA-502's workpaperId** with
  `collectionKey = ".{AREA}.RelevantAssertion"`.
- KBA-502's bulk form GET does NOT embed the RelevantAssertion child
  rows (`OverallAuditAreas[].childObjectList` returns `[]` even after refresh — the rows are cross-form
  bindings the SPA renders per area), and `inventory_form` over-filters the form (~66/68 columns dropped),
  so a GET-only read shows just `Comment` and looks like a rollup. It isn't. Write by AREA collectionKey
  against KBA-502's wpId; only relevant assertions (from KBA-400) render and are writable.
- KBA-502 also owns its FS-level risks (`FinancialLevelRisks` — Management Override etc.; only the
  row-level `Comment` is directly writable there, the rest is pt5-linked).

Mechanics: `cch-axcess` `field-conventions.md` §5.

## Write order (the judgment sequence — cch-axcess executes it)

Per scoped area, per relevant assertion (from KBA-400), against **KBA-502's wpId**:

1. **IR** = SBM / Mod / Low from `defaults/{CODE}.md` (never MAX — see below).
2. **CR** = **MAX** on a substantive audit — set it explicitly (the pre-committed state-2 default still
   raises a Question-Unanswered diagnostic). Lower CR only with controls support (the AUD-100 decision).
3. **RMM** — with CR=MAX, RMM=IR deterministically; set it explicitly to clear its QU. (If CR < MAX, read
   CCH's `recommendedAnswer` back and write that — matrix in `references/risk-framework.md`.)
4. **Planned approach** — one POST per approach checkbox.
5. `selected` is NOT required — assertion presence on the grid already reflects KBA-400's selection; N/A
   assertions render `visible:false` (Cash→AV etc., see `cascade/kba-400.md`) and are skipped.
6. **Per-area durable unit:** writes → submit (KBA-502's wpId, never empty) → refresh, before the next
   area. Verify via the diagnostics oracle, never the in-session echo (cch-axcess §5 carries the recipe).

## Setting IR (the judgment)

1. **Apply `defaults/{CODE}.md` AS-IS** for the area + audit type — the PPC-derived IR per assertion.
2. **IR is NEVER MAX, and never elevated.** A fraud risk, significant estimate, or
   known issue is addressed by the **significant-risk flag + the audit response** (Identified Risks table
   + targeted response steps), NOT by cranking IR above the default. Do not editorialize a KBA-400 fraud
   flag into an IR bump. Deviating from a default in either direction is a firm-principal call — queue it, don't apply it.
3. **Significant risks** are flagged separately (not a 5th level) — see `references/risk-framework.md`.
4. **Set IR on EVERY scoped assertion — an unset IR is the defect, not an acceptable residual.** Leaving IR
   blank on a scoped area lights "Question Unanswered" diagnostics and leaves the cascade half-done; a
   scoped area always carries an IR level. Set it — never ship a scoped area with IR unset.
5. **Document the basis for every significant area — default OR adjusted.** KBA-503 (Basis for Inherent Risk
   Assessment) carries a basis line for each scoped area, not only the ones moved off default. An
   all-default run does NOT leave KBA-503 blank — a default IR still has a basis ("PPC default for the
   area/type, no engagement fact elevates it"), and that basis holds this year. Write it as you set the
   level, don't backfill. (KBA-503's basis lines are NOT visible to the diagnostics oracle or the form GET
   — "0 diagnostics" is not proof they were authored; verify by a fresh-SPA render.)

CR stays MAX on a substantive audit. Where AUD-100 set controls testing for the area and KBA-401/403 support
it, CR may drop — and KBA-502's "link to controls testing" should point to that work.

## Planned audit approach (3 choices per assertion)

The three approaches: **Combined**, **Substantive: Analytical**, **Substantive: In-depth**. Check ≥1 per
relevant assertion. Decision (from `programs/_conventions.md`):

1. **CR < MAX** (controls tested) → **Combined** is available; check it to take controls credit.
2. **Active program has analytical steps** for the assertion → **Analytical**.
3. **Active program is tests-of-details** (most areas) → **In-depth**.

Default on a substantive, no-analytical, CR=MAX engagement: **In-depth on every relevant assertion.** Add
Analytical where the program includes analytical procedures (revenue, expenses, predictable balances). The
approach should be consistent with the steps actually pulled into the AUD-8xx program. (cch-axcess carries
the exact approach valueKey codes and confirms the write took.)

## What happens on the programs (step 4 — NOT an IR write)

On the AUD-8xx program you only **link program STEPS to assertions/risks and sign them off** — that linkage
feeds BACK to KBA-502 and clears its "Relevant Assertion Unaddressed" (RAU) diagnostics. The
RelevantAssertion row's `ProgramSteps` (C09) is propertyType-5 linked/derived — never directly writable;
RAU resolves through program-step activation + assertion affirmation + sign-off at section fan-out, not a
planning-time property write. RAU residuals on un-built programs are a **fan-out residual owned by the
responsible section owner** — enumerate them with named owners, don't fight them at planning.
