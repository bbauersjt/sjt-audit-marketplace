# KBA-502 ŌĆö Summary of Risk Assessments (the risk summary)

**Form:** KBA-502 (`dataBindingKey` KBA502). Uniform across the five non-SA titles.
**Role in the cascade:** the risk **summary**. For each relevant assertion selected on KBA-400, the firm's
**Inherent Risk (IR)**, **Control Risk (CR)**, derived **RMM**, a **link to controls testing** when CR < MAX,
and the **planned audit approach** (Combined / Substantive-Analytical / Substantive-In-depth) are assessed.
KBA-502 **rolls those up for the reviewer** ŌĆö but it is **not** where they are written (see below).

## Data model ŌĆö VERIFIED 2026-05-30 (live NFP capture)

**KBA-502 is a read-through summary ŌĆö it is NOT the write target for the per-assertion grid.** A live read
of KBA-502 carries only the 6-assertion legend, the linked AUD-100 drivers, the area registry (read-through,
no IR/CR), and KBA-502's **own** FS-level risks (Management Override etc., with their own approach
checkboxes). There is **no per-assertion `RelevantAssertion` grid on KBA-502.**

The editable per-assertion **IR / CR / RMM / planned-approach grid lives on each AUD-8xx program
workpaper**, in that area's `RelevantAssertion` collection. **Write there, not to KBA-502.** KBA-502 reads
those values through and rolls them up. So the order is: KBA-400 selects the relevant assertions ŌåÆ CCH
recommends programs ŌåÆ add the AUD-8xx programs ŌåÆ set IR/CR on the program's `RelevantAssertion`. KBA-502's
own writable content is FS-level: its FinancialLevelRisks rows.

> `cch-axcess` owns the exact field locators, valueKey codes, and write/submit/verify mechanics. This file
> states the judgment and the write **target** (the program) ŌĆö not the HTTP details.

## Write order (the judgment sequence ŌĆö cch-axcess executes it)

1. **`selected = Yes` first** for every assertion you'll touch (otherwise CCH resets IR on submit; KBA-400's
   relevant-assertion selection should already have set this ŌĆö verify via cch-axcess).
2. **Then IR** = Max / SBM / Mod / Low.
3. **CR** defaults to **MAX**; leave it unless controls were tested (the AUD-100 controls decision). Lower CR
   only with controls support.
4. **RMM** ŌĆö do **not** compute it yourself; read CCH's recommended value after IR+CR are set and write that
   back. CCH derives IR├ŚCRŌåÆRMM server-side (matrix in `references/risk-framework.md`).
5. **N/A assertions** are skipped (CashŌåÆAV, InvestmentsŌåÆCU, PPEŌåÆCU, AP/OL/EquityŌåÆRO; see `cascade/kba-400.md`).
6. Submit **after** the group of writes, not between them (cch-axcess handles the submit/commit cycle).

## Setting IR (the judgment)

1. **Start from `defaults/{CODE}.md`** for the area + audit type ŌĆö the PPC-derived IR per assertion. Starting
   points, not answers.
2. **Adjust per engagement facts** (from intake): elevate where a fraud risk, significant estimate,
   complexity, related-party exposure, or a known issue attaches to that assertion; lower only with a clear,
   documentable reason.
3. **Significant risks** are flagged separately (not a 5th level) ŌĆö see `references/risk-framework.md`. A
   significant risk lives on the Identified Risks table and typically pushes IR to SBM/MAX on the affected
   assertion(s).
4. **Document the basis** ŌĆö every non-default IR needs a rationale that holds this year. That rationale is
   KBA-503 (Basis for Inherent Risk Assessment). Write the basis as you set the level, don't backfill.

CR stays MAX on a substantive audit. Where AUD-100 set controls testing for the area and KBA-401/403 support
it, CR may drop ŌĆö and KBA-502's "link to controls testing" should point to that work.

## Planned audit approach (3 choices per assertion)

The three approaches: **Combined**, **Substantive: Analytical**, **Substantive: In-depth**. Check Ōēź1 per
relevant assertion. Decision (from `programs/_conventions.md`):

1. **CR < MAX** (controls tested) ŌåÆ **Combined** is available; check it to take controls credit.
2. **Active program has analytical steps** for the assertion ŌåÆ **Analytical**.
3. **Active program is tests-of-details** (most areas) ŌåÆ **In-depth**.

Default on a substantive, no-analytical, CR=MAX engagement: **In-depth on every relevant assertion.** Add
Analytical where the program includes analytical procedures (revenue, expenses, predictable balances). The
approach should be consistent with the steps actually pulled into the AUD-8xx program. (cch-axcess carries
the exact approach valueKey codes and confirms the write took.)
